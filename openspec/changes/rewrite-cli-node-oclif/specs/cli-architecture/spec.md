## MODIFIED Requirements

### Requirement: Modular Command Architecture
The system SHALL organize CLI commands as TypeScript oclif command modules, with the public `orodc` executable acting as the CLI entry point rather than a Bash router.

#### Scenario: Database commands are modular
- **GIVEN** database-related functionality exists
- **WHEN** user executes `orodc database mysql`, `orodc database psql`, `orodc database import`, `orodc database export`, or `orodc database cli`
- **THEN** the CLI SHALL route to the corresponding TypeScript command module
- **AND** each command module SHALL use shared environment, process, UI, and Docker Compose services instead of sourcing shell libraries

#### Scenario: Command aliases route to command modules
- **GIVEN** convenience aliases exist (`orodc mysql`, `orodc psql`, `orodc cli`)
- **WHEN** user executes an alias command
- **THEN** the oclif command registry SHALL route to the corresponding command implementation
- **AND** behavior SHALL be identical to the full command path

### Requirement: Main Script Command Routing
The public `orodc` entry point SHALL delegate routing to oclif command discovery and explicit alias definitions.

#### Scenario: Command routing delegates to modules
- **GIVEN** user executes `orodc <command> [subcommand] [args]`
- **WHEN** command matches a known route
- **THEN** oclif SHALL invoke the corresponding TypeScript command module
- **AND** all remaining arguments SHALL be passed to the command parser or pass-through handler according to the command contract

#### Scenario: No arguments shows interactive menu
- **GIVEN** user executes `orodc` with no arguments in an interactive terminal
- **WHEN** `DC_ORO_NO_MENU` is not set
- **THEN** the CLI SHALL display the interactive menu
- **AND** when the current directory is not a project, the CLI SHALL first attempt to switch to the last-used registered environment

#### Scenario: Unknown commands show error
- **GIVEN** user executes an unknown command
- **WHEN** command does not match any route or smart PHP command detection rule
- **THEN** system SHALL display error message "Unknown command: <command>"
- **AND** system SHALL suggest running `orodc help` for available commands
- **AND** exit code SHALL be 1

### Requirement: Module Structure Conventions
All command modules SHALL follow consistent TypeScript structure and shared service conventions for maintainability.

#### Scenario: Module command structure
- **GIVEN** a command module file exists
- **WHEN** module is executed by oclif
- **THEN** module SHALL define its command id, summary, flags, args, aliases, and examples using oclif conventions
- **AND** module SHALL call shared initialization services when project context is required
- **AND** module SHALL return or throw structured errors instead of calling `exit` from arbitrary helper logic

#### Scenario: Module library dependencies
- **GIVEN** a command module requires common functionality
- **WHEN** module imports dependencies
- **THEN** module SHALL import shared TypeScript services for UI messages, environment initialization, path resolution, process execution, Docker Compose operations, and prompts
- **AND** module MAY import additional focused services as needed for port allocation, source-code sync modes, CMS detection, registry management, or diagnostics

### Requirement: Main Script Responsibilities
The public CLI entry point SHALL handle only process bootstrap, oclif execution, and top-level error normalization.

#### Scenario: Main script handles version command
- **GIVEN** user executes `orodc version`
- **WHEN** version command is processed
- **THEN** the CLI SHALL read version metadata from the package or formula-generated metadata
- **AND** the CLI SHALL display the installed OroDC version
- **AND** the command SHALL exit immediately after displaying version

#### Scenario: Main script handles help command
- **GIVEN** user executes `orodc help` or `orodc man`
- **WHEN** help command is processed
- **THEN** the CLI SHALL display help generated by oclif or project documentation content
- **AND** the CLI SHALL fall back to basic help if extended documentation assets are unavailable
- **AND** the command SHALL exit immediately after displaying help

#### Scenario: Main script initializes environment
- **GIVEN** user executes a command requiring project context
- **WHEN** command is not in the no-project-required command set
- **THEN** the command module SHALL call the shared environment initialization service before executing command behavior
- **AND** the CLI SHALL allow graceful failure for menu and informational commands when not in a project

### Requirement: Command Group Organization
Related commands SHALL be organized into logical oclif command groups with subcommand routing.

#### Scenario: Database command group routing
- **GIVEN** user executes `orodc database <subcommand>`
- **WHEN** subcommand is mysql, psql, import, export, or cli
- **THEN** oclif SHALL route to the appropriate database command module
- **AND** the CLI SHALL display an error for unknown database subcommands

#### Scenario: Tests command group routing
- **GIVEN** user executes `orodc tests <subcommand>`
- **WHEN** subcommand is install, run, behat, phpunit, or shell
- **THEN** oclif SHALL route to the appropriate tests command module
- **AND** the CLI SHALL display an error for unknown tests subcommands

#### Scenario: Proxy command group routing
- **GIVEN** user executes `orodc proxy <subcommand>`
- **WHEN** subcommand is up, down, restart, or install-certs
- **THEN** oclif SHALL route to the appropriate proxy command module
- **AND** the CLI SHALL display an error for unknown proxy subcommands

### Requirement: Single-File Command Modules
Simple commands without subcommands SHALL be implemented as focused TypeScript command modules.

#### Scenario: Single-file command routing
- **GIVEN** commands like `init`, `purge`, `config-refresh`, `ssh`, `install`, `cache`, `php`, `composer`, `platform-update`, `doctor`, `env`, `status`, `conf`, `agents`, and `exec` exist
- **WHEN** user executes `orodc <command>`
- **THEN** oclif SHALL route directly to the matching command module
- **AND** no Bash module delegation SHALL be required

### Requirement: Command Aliases
The system SHALL provide convenient aliases for commonly used commands while maintaining modular oclif routing.

#### Scenario: Compose command aliases
- **GIVEN** user executes `orodc start`, `orodc stop`, `orodc restart`, `orodc up`, `orodc down`, `orodc logs`, or `orodc ps`
- **WHEN** alias command is executed
- **THEN** the CLI SHALL route to compose behavior with the appropriate subcommand
- **AND** behavior SHALL be identical to `orodc compose <subcommand>`

#### Scenario: Database command aliases
- **GIVEN** user executes `orodc mysql`, `orodc psql`, `orodc cli`, or `orodc db <subcommand>`
- **WHEN** alias command is executed
- **THEN** the CLI SHALL route directly to the corresponding database command behavior
- **AND** behavior SHALL be identical to `orodc database <subcommand>`

#### Scenario: Verbose toggle command
- **GIVEN** user executes `orodc v` or `orodc verbose`
- **WHEN** the toggle command is executed
- **THEN** the CLI SHALL enable verbose mode if it is disabled and disable it if it is enabled
- **AND** the CLI SHALL report the resulting state and exit immediately

### Requirement: Error Handling for Deprecated Commands
The system SHALL provide helpful error messages for deprecated command syntax.

#### Scenario: Old Docker Compose syntax shows migration help
- **GIVEN** user executes old syntax like `orodc build`, `orodc config`, `orodc cp`, `orodc create`, `orodc events`, `orodc images`, `orodc kill`, `orodc pull`, `orodc push`, `orodc rm`, `orodc run`, `orodc stats`, or `orodc watch`
- **WHEN** deprecated command is detected
- **THEN** system SHALL display error message "Docker Compose commands must use 'compose' prefix"
- **AND** system SHALL show old syntax and new syntax examples
- **AND** system SHALL suggest running `orodc help` for more information
- **AND** exit code SHALL be 1

### Requirement: Future Decomposition Guidelines
The system SHALL maintain guidelines for command and service decomposition in the TypeScript architecture.

#### Scenario: Current entry point size is acceptable
- **GIVEN** the public entry point only bootstraps oclif
- **WHEN** evaluating entry point size
- **THEN** current size SHALL be considered reasonable if it contains no command business logic
- **AND** command behavior SHALL remain in command modules or shared services

#### Scenario: Conditions for extracting shared services
- **GIVEN** command modules share logic
- **WHEN** two or more command modules need the same behavior
- **THEN** extraction into a shared TypeScript service SHALL be considered
- **AND** shared services SHALL expose typed inputs and outputs

#### Scenario: Conditions for command-specific extraction
- **GIVEN** a command module grows complex
- **WHEN** command logic includes multiple separable workflows
- **THEN** workflow logic SHALL be extracted into focused services or helper modules
- **AND** the command module SHALL remain responsible for parsing, orchestration, and user-facing errors

#### Scenario: Decomposition decision criteria
- **GIVEN** consideration of further decomposition
- **WHEN** evaluating whether to extract functionality from a command module
- **THEN** extraction SHALL be considered if the logic is reused by multiple commands
- **AND** extraction SHALL be considered if direct unit testing requires separating orchestration from side effects
- **AND** extraction SHALL NOT be performed solely for the sake of reducing line count
